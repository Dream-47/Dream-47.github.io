C#命名规范

2021-01-06 C#命名规范 评论

谷歌C#代码规范中文版本

参考

  • 谷歌文档
  • Nikita Starichenko

命名指导

命名规则遵循 Microsoft的C＃命名准则。如果未指定Microsoft的命名准则（例如，私有和局部变量），则规则取自 CoreFX C＃编码准则

规则摘要：

约定

  • PascalCase
    • 每个单词首字母大写例如 ABC
  • camelCase
    • 第一个单词首字母小写，其余大写例如 aBC
  • _camelCase
    • 第一个单词首字母小写，首字母增加下划线，其余大写例如 _aBC

代码

  • 类，方法，枚举，公共字段，公共属性，命名空间采用 PascalCase
  • 局部变量，参数采用 camelCase
  • privite，protected，internal，protected internal修饰的字段或者属性采用_camelCase
  • 命名准则不受修饰符的影响，例如const，static，readonly等等
  • 接口名以I作为首字母，例如IInterface
  • 不要在标识符的任意位置使用下划线

文件

  • 文件名与文件夹名采用PascalCase，例如MyFile.cs
  • 文件名应与主类名称相同
  • 通常，每个cs文件只含有一个核心类

组织

  • 多个修饰词出现时应以以下顺序排序：public protected internal private new abstract virtual override sealed static readonly extern unsafe volatile async，一般是访问权限词在前。

  • using声明位于所有命名空间顶部， 除了System的引入位于首位，其余皆以字母顺序排序

  • 类成员顺序

    • 类中成员采用以下顺序分组

      • 嵌套类，枚举，委托和事件
      • static，const与readonly 字段
      • 字段与属性
      • 构造器与终结器
      • 方法

    • 在每个组中，元素应该按照以下顺序排列

      • public
      • interna
      • protected internal
      • protected
      • private

空白规则

参考Google Java准则，

  • 每行最多一个语句

  • 每个语句最多一个任务

  • 缩进为两个空格，没有tab

  • 列数限制：100

  • 在大括号之前不能换行

  • 在右大括号和else之间没有换行

  • 当可选时也使用大括号

  • if/for/while等后面要有空格，逗号后面要有空格

它指的是字符与字符留多少空白，什么时候换行，使用VS默认配置即可

编码指导

常数

  • 变量和字段如果能使用const则使用const

  • 如果const不行，readonly可以是一个合适的选择。

  • 相对magic numbers而言，使用常量更佳

Genorators vs Container

genorators通常指的是自定义的迭代器

  • genorator代码可读性通常比container代码差，显而易见

  • 当并非所有数据要处理时，genorator代码通常更加优秀

    假设你有一个迭代功能，返回1到1000的整数，实际上你可能只需要迭代到10就够了

    使用container需要1000个单位时间与空间，而genorator只需要十个单位

    如果1000个数全部迭代，genorator代码应该更慢一点因为多个yield return

  • genorator代码通过ToList填充容器性能比直接使用container代码填充容器要差

  • genorator代码进行多次迭代时会比直接迭代container慢

  • 只有在条件2满足时选择genorator，否则选择container

属性

  • 对于单行只读属性，使用=>表达更好

  • 对于其他的，使用{ get; set; }语法

表达式主体语法

  • 在lambda与属性中明智的使用表达式

  • 不要在方法定义上使用。C＃7上线时将对此进行回顾，因为C＃7大量使用此语法。

  • 对齐方式按照VS默认值

Structs与Classes对别

  • 结构与类非常不同

    • 结构始终按值传递并返回

    • 为返回结构体对象中的成员赋值不会改变原结构体中的值

      trasform.position.x = 10;不会改变ttransform.position的值

  • 大多数使用class

  • 值类型通常有这样几个特点，类型的实例很小，并且寿命很短，通常嵌入其他实例中，具有这些特点的前提下可以考虑使用struct

  • struct相比class通常拥有更好的性能(因为他是在堆中产生，运算的)，但也不能滥用，酌情使用

Lambdas对比命名方法

  • 如果表达内容比较繁杂(例如多个语句)，或者被很多地方使用，那么使用方法更好

字段初始化

  • 通常建议字段初始化

扩展方法

ref 与 out

数组与列表

  • 一般来说，公开的变量属性，返回值用list<>
  • 当容器的长度需要改变时使用list<>
  • 容器的长度需要固定，并且其内容在构建时可知使用array
  • 多维数组选择array
  • 注意
    • array，List<>都代表线性的连续容器
    • 与C ++数组std::vector相似，array具有固定容量，List<>但可以添加
    • 在某些情况下，数组的性能更高，但通常List<>更灵活。

字符串拼接

  • 使用StringBuilder通常比直接拼接字符串更好

从容器中删除元素

  • 如果只需要满足删除某些条件的元素，建议someList.RemoveAll(somePredicate)

  • 如果需要在迭代中完成其他工作，RemoveAll可能还不够。一种常见的替代模式是在循环外部创建一个新容器，插入要保留在新容器中的项目，并在迭代结束时将原始容器与新容器交换。

调用委托

  • 当调用委托时，使用空条件判断既简洁又安全，例如SomeDelegae.?Invoke

使用var关键词

  • 当类型嘈杂，显而易见，不重要的时候，var可以提高可读性

  • 鼓励使用

    • 类型是显而易见的

    • 仅仅直接传递给其他方法的瞬时变量

  • 不鼓励使用

    • 用于基本类型

    • 内置数据类型

特性(Attributes)

  • 属性应显示在与它们关联的字段，属性或方法上方的行上，并以换行符与成员分隔。
  • 多个属性应使用换行符分隔。这样可以更轻松地添加和删除属性，并确保每个属性都易于搜索。

参数命名(Argument Naming)
